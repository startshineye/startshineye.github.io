二:Oracle数据类型

1.数字类型

number(数字类型)[包括整数类型和浮点数类型]:
number(3):表示该字段是整数类型,位数最大为三位,最大值是999
number(3,2):该字段可以出现小数，3表示有效数字的位数，2表示小数位数

案例
创建一张测试表,向其中插入数据 :t_test(id number(3,1))  
 create table t_test(id number(3,2));


1. 向表中插入数据9
 insert into t_test values(9);


2. 向表中插入数据10  
 insert into t_test values(10);


出现以上错误原是因为我们定义的id有效数字是3位,且保留两位小数,对于10会变成10.00此时有效数字位数是4位,超出了指定位数.

2.字符类型
  • char 字符类型：定长字符串，不可变长度字符串。 在Oracle数据库中：char(10)，假如传递的数据是’jack’，底层实际存储的是’jack+6个空格’
  • varchar 字符类型：可变长度字符串。
    varchar是SQL语句规范中的标准。 通用的。 Varchar默认采用一个字节编码，汉字、全角等采用两个。

  • varchar2 字符类型：可变长度字符串
    varchar2只有Oracle数据库采用， 不是通用的，但是varchar2采用默认采用两个字节编码，可以容纳更多种类的语言。 兼容性更好一些。和数据库的字符集有关系

    字符类型有两种char与varchar(varchar2)有什么区别?
CHAR存储定长字符串。
varchar,VARCHAR2存储可变长字符串

3.日期类型

  • date 日期类型

4扩充类型

  • blob Binary Large OBject(二进制大对象)
    通常存储：图片、声音、视频等二进制数据。
  • clob Character Large OBject(字符大对象)
    通常存储：大文本

三: Oracle中to date,to char,to_number解析

1 to_date解析

to_date

  1. to_date函数的作用:将”字符类型数据”转换为”日期类型”

  2. to_date函数的运算结果类型:date

  3. to_date函数的语法格式:to_date(“字符类型数据”,”日期格式”)

  4. 关于日期格式

  5. to_date函数的使用场景 

    案例:创建一张表  t_person(name varchar2(32),birthday date)
create tale t_person(name varchar2(32),birthday date);

--向表中插入数据1
insert into t_person values('zs','1987-10月-13');--ORA-01861: 文字与格式字符串不匹配

修改SQL语句
insert into t_person values('zs',to_date('1987-10月-13','YYYY-MON-DD'));

--向表中插入数据2
insert into t_person values('LS','11月-15-1988');--ORA-01861: 文字与格式字符串不匹配
修改SQL语句
insert into t_person values('LS',to_date('11月-15-1988','MON-DD-YYYY'));

--向表中插入数据3
insert into t_person values('ww','22-12月-86');//可以插入
底层会自动调用to_date函数进行转换

  6. 结论
    大多使用在插入操作,如果数据库字段定义的是date类型,但是传递的数据是字符类型
    那么需要手动调用to_date函数进行转换;除非插入的字符串的格式和oracle数据库默认的日期格式
    相同,会自动进行类型转换

    to_char

  7. to_char函数的作用:将”日期类型”数据转换为字符类型

  8. to_char函数的运算结果类型:char

  9. to_char函数的语法格式:to_char(“日期数据”,”日期格式”)

  10. 常用的数字格式

  11. to_char函数的使用场景 

    案例:查询员工的编号,姓名,雇用日期
select empno,ename,hiredate from emp;

要求:雇佣日期按照年-月-日 时:分:秒的格式进行展示
select empno,ename,to_char(hiredate,'YYYY-MON-DD HH24:MI:SS') from emp;

  12. 结论 

    大多使用在select操作中,如果默认的日期转
换格式不符合用户的要求,可以手动调用to_char函数进行转换

四: Oracle与分页相关的rownum解析

  1. rownum
    是oracle数据库特有的内容,是一个隐含字段,为每一个查询的结果集维护了一个自增的行号行号从1开始,以1递增

  2. 不同的数据库,分页的SQL不同
    mysql–>limit
    oracle–>三层嵌套select+rownum

  3. rownum都支持哪些操作?
    在进行查询的时候,行号一定要包含1
    <,<=,>0,>=1

  4. rownum和表中的记录是否存在绑定关系?
    不存在

#####——-oracle分页实例(重点)——

5.案例: 查询出员工表的前5条记录
select empno,ename from emp where rownum<=5;

######6.案例: 查询出员工工资排名在前5条记录
错误的写法: 

select empno,ename,sal 
from emp 
where rownum<=5 order by sal desc;

正确的写法: 

    第一步:先按照工资降序排序
select empno,ename,sal,rownum from emp order by sal desc;

    第二步:将上面查询的结果作为一张临时表t,取出前5条
    select 
        empno,ename,sal
    from
    (select empno,ename,sal from emp order by sal desc)t
    where
        rownum<=5;
7案例:查询出员工工资排名在前[3-9]条记录
第一步:将工资排名在前9的员工查询出来
select 
    empno,ename,sal,rownum
from
(select empno,ename,sal from emp order by sal desc)t
where
    rownum<=9;
第二步:将上面的查询结果作为一个临时表 tt
select 
    empno,ename,sal
from
    (select 
empno,ename,sal,rownum linenum
    from
(select empno,ename,sal from emp order by sal desc)t
    where
    rownum<=9)
where 
    linenum >=3;
8.通用的分页sql
每页显示3条
第1页 (0-3]
第2页 (3-6]
第3页 (6-9]
...

每页显示pageSize条
第pageNo页 ( (pageNo-1)*pageSize-pageNo*pageSize]

    select 
        tt.*
    from
        (select 
            t.*,rownum linenum
        from
            (业务SQL)t
        where
        rownum<=pageNo*pageSize)tt
    where 
        linenum >(pageNo-1)*pageSize;

正文

一:String,StringBuffer,StringBuilder之间的区别与联系

1. 区别与联系
  • [相同点]String,StringBuffer,StringBuilder 底层都是采用char数组
  • [不同点]String是定长的字符串: 是存储在方法区中的常量池之中,任何对字符串的修改操作都会创建新的对象,不适合做字符串的拼接
  • [不同点]StringBuffer,StringBuilder是可变长的
    1. StringBuffer,StringBuilder 默认的数组初始化长度是16
    2. StringBuffer,StringBuilder 如果长度不够的话,需要对数组进行扩容,底层使用的数组拷贝的方式对数组进行扩容 System.arraycopy()
    3. 该方法前面用native修饰,属于本地方法调用 (JNI Java Native Invoke),表示调用了其它语言编写的dll文件
    4. 关于优化:在做大数据量字符串拼接的时候,可以指定数组的初始化长度,减少扩容的次数,减少数组拷贝的次数
    5. StringBuilder是JDK1.5之后出现的（比较新）
      StringBuffer是JDK1.0就出现了（比较老，并发性不行，每一个方法上都有synchronized，效率较低，因为有些情况下synchronized没必要编写，例如：局部变量就不存在线程安全问题。）
2.源码
  • String源码分析
  • StringBuffer源码分析
  • StringBuilder源码分析
  • StringBuffer,StringBuilder父类源码分析

二:线程安全问题

1.发生条件
  1. 多个线程
  2. 共享资源
  3. 对共享资源涉修改操作
    以上三个条件都满足的情况下,才需要考虑线程安全的问题
2.解决方案
  • 多线程共享：堆内存、方法区内存(实例变量、静态变量会存在线程安全问题，实例变量在堆中，静态变量在方法区中)
    栈内存独立，一个线程一个栈。(结论：局部变量不存在线程安全问题，因为局部变量不共享)
  • 解决方法
    方案1：尽量使用局部变量
    方案2：如果是实例变量，我们可以让对象变成多例的也可以解决线程安全问题。
    方案3：如果是实例变量，我们还可以添加synchronized进行控制保证线程安全
    多例：耗费堆内存
    添加synchronized影响并发

三:总结

1. 字符串拼接速度及原因
  • StringBuilder>StringBuffer>String
  • 原因
    基于以上分析可以总结出以下原因:
    1. String是不可变的（final），每次改变其变量值，生成一个新的对象，然后将变量引用指向新对象；速度慢。
    2. StringBuffer直接操作对象指向的引用，无需产生新对象，速度很快；它是线程安全的，在维护多线程的同步等也会消耗一点性能。
    3. StringBuilder是jdk5之后新增的，用法与StringBuffer完全一致，但它是线程不安全，在单线程中最佳，因为其不需要维护线程的安全，因此是最快的
