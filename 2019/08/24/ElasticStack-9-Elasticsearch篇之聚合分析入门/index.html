ElasticStack-9-Elasticsearch篇之聚合分析入门

介绍Elasticsearch 聚合分析的功能，让大家了解其分类、组成，带领大家逐个了解、掌握每一个聚合 API 的使用方法和技巧，为后续 Kibana 使用打好基础。

1.聚合分析简介

1.1 什么是聚合分析?

  1. 1.搜索引擎用来回答如下问题:
    -请告诉我地址为上海的所有订单?
    -请告诉我最近1天内创建但没有付款的所有订单?

  2. 2.聚合分析可以回答如下问题:
    -请告诉我最近1周内每天的订单成交量是多少?(关注的不是订单，而是订单数量)
    -请告诉我最近1个月每天平均订单金额是多少?(关注的不是订单列表，而是订单金额)
    -请告诉我最近半年卖的最火的前5个商品是哪些?(关注的不是订单列表，而是订单里面的商品)

  3. 3.对于以上问题,假如我们没有用es时候,也是可以做的,但是对于每一个分析的话都是我们在客户端计算是非常的麻烦，es发现了这些问题,所以加上了聚合分析的能力。

  4. 4.聚合分析，英文为:Aggregation,是es除搜索功能外提供 的针对es数据做统计分析的功能。
    -功能丰富,提供Bucket、Metric、Pipeline等多种分析方式，可以满足大部分的分析需求。
    -实时性高,所有的计算结果都是及时返回,而hadoop等大数据系统一般都是T+1级别的

1.3 聚合分析的分类

  1. 1.为了便于理解,es将聚合分析主要分为如下4类
    -Buckets,分桶类型,类似SQL中的Group BY 语法。
    -Metric,指标分析类型,如计算最大值,最小值,平均值等等。
    -Pipeline,管道分析类型,基于上一级的聚合分析结果进行再分析
    -Matrix,矩阵分析类型-是es后面加入的

2.Metric聚合分析

2.1 Metric聚合分析类别

  1. 1.主要分为如下两类：
    -单值分析,只输出一个分析结果
    -min,max,avg,sum
    -cardinality(计算数目的,理解成distinct count)
    -多值分析,输出多个分析结果
    -stats,extended stats(多样分析,包含最小值,最大值)
    -percentile,percentile rank(百分位数统计)
    -top hits(取排在前面的hit列表)

8.es精准度问题

通过前面的分析可以知道es的强大之处,在本章节我们了解下es聚合分析底层东西。

8.1 Min聚合的执行流程


我们对age进行最小值的聚合分析,当es接收到查询语句后,coordinating Node会去获取每一个分片上的最小的值min value from Px，然后coordinating Node算出Min(p0,p1,p2,p3,p4)最小值返回给用户。

8.2 Terms聚合的执行流程


我们获取job前5个值,当es接收到查询语句后,coordinating Node会去获取每一个分片上的top 5 from Px，然后coordinating Node算出Top(p0,p1,p2,p3,p4)最小值返回给用户.上面执行流程中会不会有问题，Terms的执行流程并不永远准确。

8.3 Terms并不永远准确

我们以以下为例:我们拿两个节点为例:node1、node2。
p0上:a有5个 b有4个 c有3个 d有4个
p1上:a有5个 b有2个 c有3个 d有1个
当我们有一个查询条件:要求返回top3，此时在p0会返回:a、b、d p1返回:a c b。然后Coordinating Node对返回的节点进行整合、返回:Result中:a(10) b(6) d(4)。但是此时导致Miss:c(6) 原因也会在node1中没有返回c,从上面计算可以知道,Terms并不永远准确

8.4 Terms不准确的原因

  1. 1.数据分散在多shard上,Coordinnating Node无法得悉数据全貌。

    8.5 Terms不准确的解决方法

  2. 1.最简单方案:设置Shard数为1,消除数据分散的问题,但无法承载大数据量。
  3. 2.合理设置Shard_Size大小,即每次从Shard上额外多获取数据,以提升准确度。上面我们设置为4即可。

8.6 Shard_Size大小设定方法

我们之前说合理设置Shard_Size大小，但是我们该如何设置呢?

  1. 1.terms聚合返回结果中有如下两个统计值:
    -doc_count_error_upper_bound被遗漏的term可能的最大值。
    -sum_other_doc_count返回结果bucket的term外其他term的文档总数。

我们从下面看出:doc_count_error_upper_bound在分片1上最大值是4(由于p0上数量是:5、4、4所以我们可能遗漏最大值就是4)，同理p1上最大值是:2加起来是6
sum_other_doc_count:返回结果bucket的term外其他term的文档总数,这里面就只有c没有返回,总和是6

  1. 1.设定show_term_doc_count_error可以查看每个bucket无算的最大值

  1. 2.Shard_Size默认大小如下:
    -shard_size = (sizex1.5) + 10

  2. 3.通过调整Shard_Size的大小降低doc_count_error_upper_bound来提升准确度
    -增大了整体的计算量,从而降低了响应时间。
