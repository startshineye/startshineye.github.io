ElasticStack-8-Elasticsearch篇之深入了解Search的运行机制

本章会深入讲解 Search 的运行机制，比如 Query 和 Fetch 阶段具体哪些工作，分片为相关性算分带来了哪些问题。另外还会讲解排序、分页与遍历的解决方案和相关问题。

1.Query-Then-Fetch

  1. 1.Search执行的时候实际分两个步骤运作的
    -Query阶段
    -Fetch阶段
  2. 2.Query-Then-Fetch

    1.1 Query阶段

  3. 1.查询阶段如下所示:from-代表页数 size-代表每页的大小(就是分页查询中的pageNum,pageSize)
  4. 2.被选中的3个分片会分别执行查询并排序,返回from+size个文档和排序值,比如：from=10 size=10,那么每个节点查询返回的是20个文档,总共3个节点返回的是:20x3=60个文档，然后再排序后返回对应文档。这么做的原因是:es的 分片机制决定了node3没法知道真正的前20个文档在哪个分片上,所有需要全部查询，然后做一个整合，整合之前才能算出真正20个文档。

1.2 Fetch阶段

  1. 1.node3根据query阶段获取的文档id列表去对应的shard上获取文档详情数据。

4.分页与遍历-fromsize

  1. 1.es提供了3种方式来解决分页与遍历的问题
    -from/size
    -scroll
    -search_after

4.1分页与遍历-fromsize

  1. 1.最常见的分页方案
    -from 指明开始位置
    -size 指明获取总数

  2. 2.深度分页是一个经典的问题:在数据分片存储的情况下如何获取前1000个文档?
    -获取从990~1000的文档时,会在每个分片上都先获取1000个文档,然后再由Coordinating Node聚合所有分片的结果后在排序选取前1000个文档。
    -页数越深,处理文档越多,占用内存越多,耗时越长。尽量避免深度分页,es通过index_max_result_window限定最多到10000条数据

es是通过分片来存储的,以上分为了5个片来存储：p0、p1、p2、p3、p4。要查询1000个文档,需要查询出每个分片的前1000个文档,然后合并整合后返回1000个文档。因为数据是分片存储在各个节点上, Coordinating Node没有全局的概念。

5.分页与遍历-scroll

  1. 1.scroll提供了遍历文档集的api,以快照的 方式来避免深度分页的问题

    -尽量不要使用复杂的sort条件,使用_doc最高效
    -使用稍微复杂
