一:String,StringBuffer,StringBuilder之间的区别与联系

1. 区别与联系
  • [相同点]String,StringBuffer,StringBuilder 底层都是采用char数组
  • [不同点]String是定长的字符串: 是存储在方法区中的常量池之中,任何对字符串的修改操作都会创建新的对象,不适合做字符串的拼接
  • [不同点]StringBuffer,StringBuilder是可变长的
    1. StringBuffer,StringBuilder 默认的数组初始化长度是16
    2. StringBuffer,StringBuilder 如果长度不够的话,需要对数组进行扩容,底层使用的数组拷贝的方式对数组进行扩容 System.arraycopy()
    3. 该方法前面用native修饰,属于本地方法调用 (JNI Java Native Invoke),表示调用了其它语言编写的dll文件
    4. 关于优化:在做大数据量字符串拼接的时候,可以指定数组的初始化长度,减少扩容的次数,减少数组拷贝的次数
    5. StringBuilder是JDK1.5之后出现的（比较新）
      StringBuffer是JDK1.0就出现了（比较老，并发性不行，每一个方法上都有synchronized，效率较低，因为有些情况下synchronized没必要编写，例如：局部变量就不存在线程安全问题。）

二:线程安全问题

1.发生条件
  1. 多个线程
  2. 共享资源
  3. 对共享资源涉修改操作
    以上三个条件都满足的情况下,才需要考虑线程安全的问题
2.解决方案
  • 多线程共享：堆内存、方法区内存(实例变量、静态变量会存在线程安全问题，实例变量在堆中，静态变量在方法区中)
    栈内存独立，一个线程一个栈。(结论：局部变量不存在线程安全问题，因为局部变量不共享)
  • 解决方法
    方案1：尽量使用局部变量
    方案2：如果是实例变量，我们可以让对象变成多例的也可以解决线程安全问题。
    方案3：如果是实例变量，我们还可以添加synchronized进行控制保证线程安全
    多例：耗费堆内存
    添加synchronized影响并发

三:总结

1. 字符串拼接速度及原因
  • StringBuilder>StringBuffer>String
  • 原因
    基于以上分析可以总结出以下原因:
    1. String是不可变的（final），每次改变其变量值，生成一个新的对象，然后将变量引用指向新对象；速度慢。
    2. StringBuffer直接操作对象指向的引用，无需产生新对象，速度很快；它是线程安全的，在维护多线程的同步等也会消耗一点性能。
    3. StringBuilder是jdk5之后新增的，用法与StringBuffer完全一致，但它是线程不安全，在单线程中最佳，因为其不需要维护线程的安全，因此是最快的
