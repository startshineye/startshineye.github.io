ElasticStack-17-Beats篇之Packetbeat

本章会介绍 Packetbeat 的功能和使用技巧，带领大家用 Packetbeat 来收集网络数据并进行分析，让大家对 Packetbeat 有一个直观的感受。

16-1 1-简介

1.简介

  1. 1.Packetbeat功能一:抓取网络包数据
  2. 2.Packetbeat功能二:自动解析网络包协议,比如:
    -ICMP DNS协议
    -HTTP协议
    -Mysql/PgSQL/MongoDB协议
    -Memcache协议
    -Thrift协议
    -TLS协议

解析这些协议的好处是:抓到了这些包,看到的不仅仅是二进制码,而是人力可读的一些信息。然后针对这些信息做一些针对性性能分析。还有一些故障定位。

2.抓包

  1. 1.说起抓包,常见的抓包工具如下:
    -tcpdump(服务器端抓包)
    -wireshark
    上面底层都是基于libpcap实现。

  2. 2.Packetbeat抓包底层依赖主要有两种:
    -pcap基于libpcap实现,跨平台支持
    -af_packet仅支持linux系统,基于内存映射的嗅探技术,性能更好:

    上面buffer_size_mb就是值内存大小,内存设置的越大,性能就越高。

  3. 3.Packetbeat配置协议
    我们在packetbeat.yml配置文件中配置:
    配置时候,我们把要解析的协议都写上,Packetbeat在抓取网络包时候,就会按照上面协议来解析了。

  4. 4.network flows
    -抓取记录网络流量数据,不涉及协议解析。有了流量数据,就可以做分析了。比如看节点a,b,c三者间流量的数据情况就可以用flows
    -flows的配置主要是两个参数:timeout和period(timeout:在超时时间内如果还没有收到数据包,我们认为发送的数据已经结束了，period每隔一段时间统计一次,让其有一个时间段的记录)

  5. 5.timeout和period的区别

    –由上图可以知道,数据字段中有一个final的字段,数据包结束了的,其对应的final就是为true。所以我们在最终分析时候只要过滤final为true的，才是正确的流量。period对应的final永远为false。
    –由上面可以知道,fows生成的数据有flow_id指明唯一性。
    –source指明数据包的来源:里面有ip和port以及mac地址。stats表明source给dest发送了一个数据包,包大小为85字节。

    –dest指明数据包的目的地址:同样包含ip、port和mac地址。stats表明dest给source返回了一个数据包,包大小为135字节。

ElasticStack-16-Beats篇之Metricbeat

本章会介绍 Metricbeat 的功能和使用技巧，让大家对 Metricbeat 的使用有一个直观的感受。

16-1 -简介

1. Metricbeat主要功能:

  1. 1.定期收集操作系统、软件或服务的指标(Metricbeat)数据
  2. 2.存储在Elasticsearch中进行实时的分析

    2. 什么是Metrics

    我们把Metrics和Logs做一个比较:
  3. 1.Logs
    -用于记录”离散的事件”,具有随机性。例如:应用程序的调试信息或错误信息等。
  4. 2.Metrics
    -用于记录”度量”或”可聚合的数据”,具有计划性。例如,服务的响应时长等。
  5. 3.日志和Metrics的比较

    -从上面我们可以看到,logs是没有规律的,而Metrics是有规律的,每10分钟收集1次。

    3.Metricbeat组成

    Metricbeat有2个主要的概念:Module、Metricset
  6. 1.Module
    -和我们之前说的概念类似:Metricbeat把要收集的数据已经封装好了,开箱即用,Module即是:Metricbeat收集指标的对象,比如:linux、windows、mysql等。
  7. 2.Metricset
    -Metricbeat收集指标集合,该集合以减少收集指标调用次数为划分依据。1个Module可以有多个Metricset。

    4.实例

    以下我们看一下Redis Module(从redis里面收集数据)

    -Redis Module就是从redis里面获取一些基本信息,熟悉redis的朋友知道:redis有一个INFO的命令对应的Redis Module里面有一个InfoMetricset
    -Redis Module抓了数据就把Metrics扔到es中去,再通过kibana进行数据可视化分析。

ElasticStack-15-Beats篇之Filebeat

本章会介绍 Beats 的作用和组成，然后为大家详细介绍 Filebeat 的功能和常见配置，同时会详细讲解如何使用 Module 模块来快速完成日志的收集到分析工作。

15-1 beats简介

1.简介


1.beats是一个轻量级的传输者(与logstash相比较)
2.采用golang语言开发,性能很高。
3.部署便捷,直接执行编译好的可执行文件即可。logstash这种基于jvm的,启动慢、占用资源比较多。
4.因为beats是轻量级,所以日志收集时候比较容易,所以beats有大量的数据收集的种类。基本上所有可以操作的数据都可以通过beats收集到elasticsearch上。(beats极大的丰富了ElasticStack可以收集的数据源)。

2.架构


以上是beats常用的架构图。
1.PacketBeat是beats家族中第一个产品,他主要收集网络包的(Wire Data),互联网时候都是通过网络传输的,通过底层抓包形式实现。可以获取到任意想要获取的网络数据。从而监控网络、系统性能，查找问题。
2.FileBeat就是收集日志的。
3.MetricBeat是手机指标数据的。
4.WinlogBeat是处理windows日志的。
5.以上beat都有一个libBeat的底层,然后数据传送到elasticsearch或者通过logstash传送到elasticsearch。
6.为什么beats需要经过logstash呢?beats主要是做数据收集的,他没有logstash数据处理那么强大的功能。

15-4 Filebeat常见架构及ingest_node介绍

Filebeat架构一


filebeat采集日志->将日志发送到logstash(logstash做一些日志复杂处理:比如:grok解析等)–>然后logstash将查询的日志放在es中存储、查询。–>最后kibana做可视化展示

Filebeat架构二


fileeat收集日志—->elasticseach的Ingest Node(具有数据处理能力)—>kibana做数据分析

  1. 1.Elasticsearch Ingest Node
    5.x新增的一个节点
    -在数据写入es前(bulk/index操作)对数据进行处理
    -可设置独立的ingest node专门进行数据转换处理 node.ingest:true
    -api endpoint为pipeline
  2. 2.Pipeline由一系列processor组成,类似于logstash的filter plugin,数据处理完之后就会流转到es中

  3. 3.Pipeline的API比较简单,主要有如下4个:
    -创建PUT
    -获取GET
    -删除DELETE
    -模拟调试SIMULATE(必须重点掌握的api)

  4. 4.PUT/GET/DELETE处理

  5. 5.SIMULATE

    a.一种是没有创建pipeline的时候,通过pipeline的参数指定(把具体pipeline的内容放到里面),然后通过docs参数作为模拟测试(把要测试的文档放在source里面)。
    b.如果pipeline已经创建的话(比如我们已经创建了test的pipeline),我们可以在后面接入:_simulate作为测试,通过docs指定文档作为测试。

3.1计算机中的数据表示方法

  1. 1.计算机中最小的单位叫做:字节(Byte)
  2. 2.1个字节是8个二进制位:1Byte = 8bit

  3. 3.为什么我们需要用“二进制”表示数据。由于我们现在使用的是电子计算机。电子计算机是逻辑电路元件做成的。因为计算机中是用电的,我们的电流状态只有2钟状态。一种是高电位,高电位对应的电路会通,我们接通一个灯泡后会亮,那么灯亮了就是1；低电位,电路不通,此时接入一个灯泡后灯不会亮。那么认为低电位时候,电流就是0。

  4. 4.高电位是1,低电位是0,那么计算机处理再复杂的数据也脱离不了这些1和0；所以不管我们听歌、看电影、玩游戏等。在计算机中由cpu处理的数据全都是101010等组成。人对于这些数据是不敏感的,人和计算机进行交互,其实就是十进制和二进制进行变换。

  5. 5.在计算机中如果全部用二进制表示将会有很多很多1010…，挺麻烦，所以引入了十六进制。 十进制、二进制、十六进制之间表示关系如下:
  6. 6.计算机计算时候是按照二进制,展示给人看的时候是十进制。我们编程时候使用十六进制。

3.2内存管理

知道计算机的数据表示方法后,那我们需要知道我们的内存到底是什么呢?

  1. 1.内存由操作系统统一管理。
  2. 2.一个字节是8个二进制位,我们在计算几种插入2个内存条,4个内存条。插一根4G内存条和2根2G内存条其实是等效的。计算机中不管插入了多少内存条，他会把内存看成一个整体来计算内存大小。
  3. 3.但是计算机并不是插多少根内存条就可以的,比如:32位的计算机不管插入多少内存条,最大使用内存是4G。
  4. 4.为什么32位操作系统只能使用4G内存呢?因为32位的电脑上,cpu的地址总线是32位。也就是操作系统的寻址空间是32位。32位指的是:给内存编号只能编到32个二进制位。这个编号就是类似于我们接到的门牌号码一样。地址总线有多少根那么存在状态就有对应的组合。每一根总线有2种状态。32根总线就有2的32此房个状态。其中一个编号就能代表计算机中内存的最小存储单位字节。则可以存储2的32次方的字节。
  5. 5.2的32次方个字节等于多少呢?
  6. 6.64为计算机存储2的64位,此时64位置系统下的内存足够使用了。
  7. 7.编号=唯一的内存字节的地址。
  8. 8.应用程序由操作系统来调用的。

  9. 9.main()函数就是所有函数的入口。

  10. 10.计算机内存分布情况如下:

    a.低位内存方便管理又进行了分片:最低位的 代码段(c语言代码-函数编译之后存到磁盘,一旦运行此程序后,代码编译后的二进制数据加载到内存中,此部分内存就是代码段)。
    b.声明的一些全局变量或者声明的一些常量:const int i;int global;这些存放到数据段。
    c.一般数据段内存地址编号通常会比代码段内存大小要大,代码段和数据段都是我们操作系统预先规划好的，不管有多少应用程序,代码段和程序段都是在低位的。
    d.中间绿色区域为:自由可以分配的内存。此区域越大调用的资源就越多。
    e.除了以上内存外:计算机还有堆内存、栈内存。

ElasticStack-12-Logstash篇之入门与运行机制

1.入门及架构简介

1.1 Logstash介绍

  1. 1.数据收集处理引擎
  2. 2.ETL工具

1.2 架构简介

1.左边是各种各样的数据源:文件、excle、http的service、数据库、自定义数据源
2.右边是logstash支持的功能:Analysis(指标分析)、Archiving(归档)、es、Monitoring(监控)、Alerting(报警)

logstash分为3个阶段:
1.Input:数据采集
2.Filter:数据解析/转换
3.数据输出


logstash处理流如下:
数据从标准输入流(stdin)进来、然后经过filter插件:(grok、date、geoip、useragent)、执行完之后进入output阶段、此时输出到elasticsearch

1.3 名词解析

  1. 1.Pipeline
    -input -filter-output的3阶段处理流程
    -队列管理(Pipeline中涉及到一个队列,自己写pipeline插件的话也需要会写pipeline生命周期的管理)
    -插件生命周期

  2. 2.Logstash Event
    -内部流转的数据表现形式
    -原始数据在input被转换为Event,在output event被转换为目标格式数据
    -在配置文件中可以对Event中属性进行增删改查


原始数据从Input进来之后,会经过decode,将原始数据转换成Event、输出时候在Output中通过encode将Event转换你为Data。

我们将codec-decode加入后变成如下形式:

1.原始数据通过Input中的codec-decode把数据转换成Logstash Event、经过filter中各个plugin的处理后还是Logstash Event、再经过Output中的codec-encode之后转换为原始数据。

2.Event的生命周期

2.1 整个架构

接下来我们整体了解下:logstash的整个架构。

1.1.每个input可以有哦多个的,每个input绑定了一个codec.
2.2.箭头”->”表示的是数据流向
3.3.通过input进来的数据经过codec后进行queue中,然后queue把数据分发到不同的pipeline中,pipeline由:Batcher、Filter、Output组成。
4.4.Batcher作用是批量从Queue中取数据(每次取多少数据是可以配置的)。
5.5.上面的架构是logstash经过很多次迭代后形成的架构。

2.2 一个logstash的整个生命历程(life)

  1. 1.首先input会从web.log中收集数据(Line1,line2,..代表的是每一行数据)

  2. 2.Json Codec读取到line1数据之后会将其转换成Event

  3. 3.Event经过Queue后流转到不同的pipeline里面的batcher里面,经过时间推移,batcher里面缓存到越来越多的数据,数据达到阈值或者时间到了之后经过filter后经过output、最后输出到其他中间件,输出之后发送ACK给Queue。

3.Queue简介

3.1 Queue分类

  1. 1.In Memory
    -无法处理进程Crash、机器宕机等情况,会导致数据丢失。
  2. 2.Persistent Queue In Disk
    -可处理进程Crash等情况,保证数据不丢失。
    -保证数据至少消费一次。
    -充当缓冲区,类似于kafaka消息队列，可以替代kafaka等消息队列的作用。

3.2 Persistent Queue容灾特性

  1. 1.数据从Input后到达PQ(Persistent Queue)、然后数据会在DISK上一个备份、然后DISK通知PQ我已经收到这个数据了。并返回给Input、之后数据就会到Filter/Output、然后Output收到数据并推送出去之后发送消息给PQ(ack)、然后通知DISKs删除数据。

4.线程简介

在之前的架构中、logstash有哪些线程

我们一般进行logstash调优时候一般主要是对线程数进行调节。

4.1 线程相关配置

  1. 1.pipeline.workers | -w
    -pipeline线程数,即filter_output的处理线程数,默认是cpu核数。

  2. 2.pipeline.batch.size | -b
    -Batcher一次批量获取的待处理文档数,默认是125,可以根据输出进行调整,越大会占用越多的heap空间,可以通过jvm.options调整

  3. 3.pipeline.batch.delay | -u
    -Batcher等待的时长,单位为ms,默认是50ms (在指定时间内还没有达到batcher的容联,也会把数据送给Filter/Output处理)
