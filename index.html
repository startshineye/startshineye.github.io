15-1 beats简介

1.简介


1.beats是一个轻量级的传输者(与logstash相比较)
2.采用golang语言开发,性能很高。
3.部署便捷,直接执行编译好的可执行文件即可。logstash这种基于jvm的,启动慢、占用资源比较多。
4.因为beats是轻量级,所以日志收集时候比较容易,所以beats有大量的数据收集的种类。基本上所有可以操作的数据都可以通过beats收集到elasticsearch上。(beats极大的丰富了ElasticStack可以收集的数据源)。

2.架构


以上是beats常用的架构图。
1.PacketBeat是beats家族中第一个产品,他主要收集网络包的(Wire Data),互联网时候都是通过网络传输的,通过底层抓包形式实现。可以获取到任意想要获取的网络数据。从而监控网络、系统性能，查找问题。
2.FileBeat就是收集日志的。
3.MetricBeat是手机指标数据的。
4.WinlogBeat是处理windows日志的。
5.以上beat都有一个libBeat的底层,然后数据传送到elasticsearch或者通过logstash传送到elasticsearch。

3.1计算机中的数据表示方法

  1. 1.计算机中最小的单位叫做:字节(Byte)
  2. 2.1个字节是8个二进制位:1Byte = 8bit

  3. 3.为什么我们需要用“二进制”表示数据。由于我们现在使用的是电子计算机。电子计算机是逻辑电路元件做成的。因为计算机中是用电的,我们的电流状态只有2钟状态。一种是高电位,高电位对应的电路会通,我们接通一个灯泡后会亮,那么灯亮了就是1；低电位,电路不通,此时接入一个灯泡后灯不会亮。那么认为低电位时候,电流就是0。

  4. 4.高电位是1,低电位是0,那么计算机处理再复杂的数据也脱离不了这些1和0；所以不管我们听歌、看电影、玩游戏等。在计算机中由cpu处理的数据全都是101010等组成。人对于这些数据是不敏感的,人和计算机进行交互,其实就是十进制和二进制进行变换。

  5. 5.在计算机中如果全部用二进制表示将会有很多很多1010…，挺麻烦，所以引入了十六进制。 十进制、二进制、十六进制之间表示关系如下:
  6. 6.计算机计算时候是按照二进制,展示给人看的时候是十进制。我们编程时候使用十六进制。

3.2内存管理

知道计算机的数据表示方法后,那我们需要知道我们的内存到底是什么呢?

  1. 1.内存由操作系统统一管理。
  2. 2.一个字节是8个二进制位,我们在计算几种插入2个内存条,4个内存条。插一根4G内存条和2根2G内存条其实是等效的。计算机中不管插入了多少内存条，他会把内存看成一个整体来计算内存大小。
  3. 3.但是计算机并不是插多少根内存条就可以的,比如:32位的计算机不管插入多少内存条,最大使用内存是4G。
  4. 4.为什么32位操作系统只能使用4G内存呢?因为32位的电脑上,cpu的地址总线是32位。也就是操作系统的寻址空间是32位。32位指的是:给内存编号只能编到32个二进制位。这个编号就是类似于我们接到的门牌号码一样。地址总线有多少根那么存在状态就有对应的组合。每一根总线有2种状态。32根总线就有2的32此房个状态。其中一个编号就能代表计算机中内存的最小存储单位字节。则可以存储2的32次方的字节。
  5. 5.2的32次方个字节等于多少呢?
  6. 6.64为计算机存储2的64位,此时64位置系统下的内存足够使用了。
  7. 7.编号=唯一的内存字节的地址。
  8. 8.应用程序由操作系统来调用的。

  9. 9.main()函数就是所有函数的入口。

  10. 10.计算机内存分布情况如下:

    a.低位内存方便管理又进行了分片:最低位的 代码段(c语言代码-函数编译之后存到磁盘,一旦运行此程序后,代码编译后的二进制数据加载到内存中,此部分内存就是代码段)。
    b.声明的一些全局变量或者声明的一些常量:const int i;int global;这些存放到数据段。
    c.一般数据段内存地址编号通常会比代码段内存大小要大,代码段和数据段都是我们操作系统预先规划好的，不管有多少应用程序,代码段和程序段都是在低位的。
    d.中间绿色区域为:自由可以分配的内存。此区域越大调用的资源就越多。
    e.除了以上内存外:计算机还有堆内存、栈内存。

  3. 3.Event经过Queue后流转到不同的pipeline里面的batcher里面,经过时间推移,batcher里面缓存到越来越多的数据,数据达到阈值或者时间到了之后经过filter后经过output、最后输出到其他中间件,输出之后发送ACK给Queue。

3.Queue简介

3.1 Queue分类

  1. 1.In Memory
    -无法处理进程Crash、机器宕机等情况,会导致数据丢失。
  2. 2.Persistent Queue In Disk
    -可处理进程Crash等情况,保证数据不丢失。
    -保证数据至少消费一次。
    -充当缓冲区,类似于kafaka消息队列，可以替代kafaka等消息队列的作用。

3.2 Persistent Queue容灾特性

  1. 1.数据从Input后到达PQ(Persistent Queue)、然后数据会在DISK上一个备份、然后DISK通知PQ我已经收到这个数据了。并返回给Input、之后数据就会到Filter/Output、然后Output收到数据并推送出去之后发送消息给PQ(ack)、然后通知DISKs删除数据。

2 写性能优化

2.1 ES写数据过程

ES写数据过程是:refresh、translog、flush。
接下来具体看每一个过程的作用

2.1.1 ES写数据-refresh

  1. 1.segment写入磁盘的过程依然很耗时,可以借助文件系统缓存的特性,先将segment在缓存中创建并开放查询来进一步提升实时性,该过程在es中被称为:refresh。

  2. 2.在refresh之前文档会先存储在一个index buffer中,refresh时将buffer中的所有文档清空并 生成segment

  3. 3.es默认每1秒执行一次refresh,因此文档的实时性被提高到1秒,这也是es被称为近实时(Near Real Time)的原因

2.1.2 ES写数据-translog
  1. 1.如果在内存中的segment还没有写磁盘前发生宕机,那么其中的文档就无法恢复了,如何解决这个问题?
    -es引入translog机制,写入文档到buffer时,同时将该操作写入translog。
    -translog文件会即是写入磁盘(fsync),6.x默认每个请求都会落盘,可以修改为每5秒写一次,这样风险便是丢失5秒内的数据,相关配置为:index.translog.*
    -es启动时会检查translog文件,并从中恢复数据。
2.1.3 ES写数据-flush
  1. 1.flush负责将内存中的segment写入磁盘,主要做如下的工作:
    -a.将translog写入到磁盘
    -b.将index buffer清空,其中的文档生成一个新的segment,相当于一个refresh操作
    -c.更新commit point并写入磁盘
    -d.执行fsync 操作,将内存中的segment写入磁盘
    -e.删除旧的translog文件

2.2 ES写性能优化

2.2.1 目的
  1. 1.目标是增大写吞吐量-EPS(Event Per Second)越高越好。
  2. 2.优化方案
    -客户端:多线程写,批量写
    -ES:在”高质量数据建模”的前提下,主要是在refresh、translog和flush之间做文章。
2.2.2 写性能优化-refresh
  1. 1.目标为降低refresh的频率
    -增大refresh_interval,降低实时性,以增大一次refresh处理的文档数,默认是1s,设置为-1直接禁止自动refresh
    -增大index buffer size,参数为indices.memory.index_buffer_size(静态参数,需要设定在elasticsearch.yxml中),默认为10%。
2.2.3 写性能优化-translog
  1. 1.目标是降低translog写磁盘的频率,从而提高写速率,但会降低容灾能力
    -index.translog.durability设置为async,index.translog.sync_interval设置需要的大小,比如120s,那么translog会改为每120s写一次磁盘
    -index.translog.flush_threshold_size默认为512mb,即translog超过该大小时会触发一次flush,那么调大该大小可以避免flush的发生。
2.2.4 写性能优化-flush
  1. 1.目标为降低flush的次数,在6.x可优化的点不多,多为es自动完成

1.1 什么是数据建模?

  1. 1.英文为 Data Modeling,为创建数据模型的过程
  2. 2.数据建模(Data Model)定义:
    -对现实世界进行抽象描述的一种工具和方法
    -通过抽像的实体及实体之间联系的形式去描述业务规则,从而实现对现实世界的映射。

1.2 数据建模的过程

  1. 1.概念模型(10%)—确定需求
    -确定系统的核心需求和范围边界,设计实体和实体间的关系。不会讨论具体的属性的。
  2. 2.逻辑模型—根据需求,设计实体关系
    -进一步梳理业务需求,确定每个实体的属性、关系和约束等。
  3. 3.物理模型—确认使用哪个数据库产品做
    -结合具体的数据库产品,在满足业务读写性能等需求的前提下确定最终的定义。
    -Mysql、MongoDB、elasticsearch等。
    -第三范式
    物理模型解决的问题其实是最终需要落地的东西。

1.3 数据建模的意义


1.首先确认你的业务
2.其次你的系统数据支撑你的业务
3.系统数据需要怎样存储,就是需要考虑到数据模型了

2.1 ES的数据建模

b.根据部署/性能需求产生物理模型,es中物理模型有ES Index Template、ES Index Mapping配置

2.2 Mapping字段的相关设置

  1. 1.enabled
    -true|false
    -当字段仅存储,不做搜索或聚合分析,把这个值设置为false,默认为true,设置为false,节省磁盘空间，比如我们做外部系统请求分析时候、有一些cookie、session数据一般不做查询分析,此时把设置为false

    -true|false
    -设置是否构建倒排索引

    -如果构建倒排索引的话,需要存储哪些信息。

    -是否存储归一化相关参数,如果字段仅用于过滤和聚合分析,可关闭

  5. 5.doc_values
    -true| false
    -是否启用doc_values,用于排序和聚合分析时候的场景

  6. 6.field_data
    -false|true
    -是否为text类型启用fielddata,实现排序和聚合分析

  7. 7.store
    -false|true
    -是否存储此字段值,默认为false,因为默认存放在source下

ElasticStack-9-Elasticsearch篇之聚合分析入门

介绍Elasticsearch 聚合分析的功能，让大家了解其分类、组成，带领大家逐个了解、掌握每一个聚合 API 的使用方法和技巧，为后续 Kibana 使用打好基础。

1.聚合分析简介

1.1 什么是聚合分析?

  1. 1.搜索引擎用来回答如下问题:
    -请告诉我地址为上海的所有订单?
    -请告诉我最近1天内创建但没有付款的所有订单?

  2. 2.聚合分析可以回答如下问题:
    -请告诉我最近1周内每天的订单成交量是多少?(关注的不是订单，而是订单数量)
    -请告诉我最近1个月每天平均订单金额是多少?(关注的不是订单列表，而是订单金额)
    -请告诉我最近半年卖的最火的前5个商品是哪些?(关注的不是订单列表，而是订单里面的商品)

  3. 3.对于以上问题,假如我们没有用es时候,也是可以做的,但是对于每一个分析的话都是我们在客户端计算是非常的麻烦，es发现了这些问题,所以加上了聚合分析的能力。

  4. 4.聚合分析，英文为:Aggregation,是es除搜索功能外提供 的针对es数据做统计分析的功能。
    -功能丰富,提供Bucket、Metric、Pipeline等多种分析方式，可以满足大部分的分析需求。
    -实时性高,所有的计算结果都是及时返回,而hadoop等大数据系统一般都是T+1级别的

  1. 1.为了便于理解,es将聚合分析主要分为如下4类

    -Matrix,矩阵分析类型-是es后面加入的

8.3 Terms并不永远准确

我们以以下为例:我们拿两个节点为例:node1、node2。
p0上:a有5个 b有4个 c有3个 d有4个
