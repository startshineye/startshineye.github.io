11 文件服务器概览
11.1 普通文件上传
  1. 文件上传,假如 我们的复杂系统被拆分成了:订单系统,用户系统,报表系统
    各自系统都会上传到各自对应的文件服务器中,他们部署在同一台服务器上是没有任何问题的:2个系统都可以互相查看文件
  2. 3个系统分开部署到不同服务器,由于3个系统彼此隔开,网络不通的情况下,用户查看订单系统时候,可能访问不到

11.2 使用文件服务器上传
  1. 3个系统会把文件统一的传到文件服务器中,不管用户在任何系统,都可以访问到这个文件
  2. 文件服务器:云服务文件系统(比如:七牛云) 自己搭建文件服务器系统(fastdfs)

11.3 fastdfs

11.4 fastdfs文件上传流程
  1. 分为:客户端 TrackerServer StorageServer
  2. tracker:跟踪器主要做调度工作，在访问上起负载均衡的作用。相当于一个项目经理
  3. storage:存储节点,存储节点存储文件，完成文件管理的所有功能。相当于一个程序员
  4. tracker和storage关系:storage会在定时的一段时间内,向tracker定时汇报一些状态(类似于程序员向项目经理汇报任务状态)
  5. 简单流程:1.storage定时向tracker汇报状态 2.客户假如有一个上传需求,此时会想tracker提出此需求 3.tracker检查是否有可用的storage(项目经理询问是否有空闲的程序员) 4.有storage,向客户端(甲方)返回可用的storage 5.客户端收到有可用的storage后作出文件上传操作 6.storage把文件存储到服务器之内 7.保存完毕之后,会返回相对路径给客户端 8.客户端做 存储文件路径 便于下次访问

 public class QRCodeUtils {

       public void createQRCode(String filePath, String content) {
           int width=300;              //图片的宽度
           int height=300;             //图片的高度
           String format="png";        //图片的格式
   //        String content="风间影月";     //内容

           /**
            * 定义二维码的参数
            */
           HashMap hints=new HashMap();
           hints.put(EncodeHintType.CHARACTER_SET,"utf-8");    //指定字符编码为“utf-8”
           hints.put(EncodeHintType.ERROR_CORRECTION,ErrorCorrectionLevel.M);  //指定二维码的纠错等级为中级
           hints.put(EncodeHintType.MARGIN, 2);    //设置图片的边距

           /**
            * 生成二维码
            */
           try {
               BitMatrix bitMatrix=new MultiFormatWriter().encode(content, BarcodeFormat.QR_CODE, width, height,hints);
               Path file=new File(filePath).toPath();
               MatrixToImageWriter.writeToPath(bitMatrix, format, file);
           } catch (Exception e) {
               e.printStackTrace();
           }
       }

       public String getContentFromQRCode(String filePath) {
           MultiFormatReader formatReader=new MultiFormatReader();
           File file=new File(filePath);
           BufferedImage image;
           try {
               image = ImageIO.read(file);
               BinaryBitmap binaryBitmap=new BinaryBitmap(new HybridBinarizer
                                       (new BufferedImageLuminanceSource(image)));
               HashMap hints=new HashMap();
               hints.put(EncodeHintType.CHARACTER_SET,"utf-8");    //指定字符编码为“utf-8”
               Result result=formatReader.decode(binaryBitmap,hints);
               return result.toString();
           } catch (Exception e) {
               e.printStackTrace();
               return null;
           }
       }
   }

1.Zookeper

zk采用多节点部署,每一个节点叫做znode,每一个节点的路径结构是唯一确定的

Znode，利用zk的短暂Znode和持久化的Znode也有属性来实现监控功能。

2 监控结构

  1. client1 client2 … 代表我们的爬虫服务
  2. 爬虫生命周期监控:每启动一个爬虫服务client时候 我们向zookeeper节点(GroupMembers)注册一个临时节点,一旦节点挂掉和停止了,临时节点(client1)就会消失,此时我们只需要一个watcher来监控所有节点,只要有节点消失掉我们watcher就会知道
  3. 正常情况下,临时节点不会消失,只有当在有一个节点挂掉了,临时节点消失,我们可以触发监控程序,知晓哪个节点挂掉了
