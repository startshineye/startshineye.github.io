1 生产环境部署建议

1.1 系统设置要到位

  1. 1.遵循官方建议设置所有系统参数
    查看如下文档:

  2. 2.参见文档”Setup Elasticsearch->Important System Configuration”
    在部署集群之前一定要把操作系统配置设定好,es给定了以下5点设置

    a.设置堆栈大小
    b.关闭swapping
    c.文件描述
    d.虚拟内存
    e.多线程设置

1.2 ES设置尽量简洁

  1. 1.elasticsearch.yml中尽量只写币贝的参数,其他可以通过api动态设置的擦书都通过api来设定。
  2. 2.参见文档”Setup Elasticsearch->Important Elasticsearch Configuration”
  3. 3.随着ES的版本升级,很多网络流传的配置参数已经不再支持,因此不要随便复制别人的集群配置参数。

1.3 elasticsearch.yml中建议设定的基本参数

  1. 1.cluster.name
  2. 2.node.name
  3. 3.node.master/node.data/node.ingest
  4. 4.network.host设置为内网ip的原因是:es本身并没有权限管理,es只要暴露了访问地址,任何人员都可以访问,从而降低了es的安全性。
  5. 5.discovery.zen.ping.unicast.hosts设定集群其他节点地址
  6. 6.discovery.zen.minimum_master_nodes一般设定为2
  7. 7.path.data/path.log
  8. 8.除上述参数外再根据需要增加其他的静态配置参数。

1.4 关于JVM内存设定

  1. 1.不要超过31GB。

  2. 2.预留一半内存给操作系统,用来做文件缓存。

2 写性能优化

2.1 ES写数据过程

ES写数据过程是:refresh、translog、flush。
接下来具体看每一个过程的作用

2.1.1 ES写数据-refresh

  1. 1.segment写入磁盘的过程依然很耗时,可以借助文件系统缓存的特性,先将segment在缓存中创建并开放查询来进一步提升实时性,该过程在es中被称为:refresh。

  2. 2.在refresh之前文档会先存储在一个index buffer中,refresh时将buffer中的所有文档清空并 生成segment

  3. 3.es默认每1秒执行一次refresh,因此文档的实时性被提高到1秒,这也是es被称为近实时(Near Real Time)的原因

2.1.2 ES写数据-translog
  1. 1.如果在内存中的segment还没有写磁盘前发生宕机,那么其中的文档就无法恢复了,如何解决这个问题?
    -es引入translog机制,写入文档到buffer时,同时将该操作写入translog。
    -translog文件会即是写入磁盘(fsync),6.x默认每个请求都会落盘,可以修改为每5秒写一次,这样风险便是丢失5秒内的数据,相关配置为:index.translog.*
    -es启动时会检查translog文件,并从中恢复数据。
2.1.3 ES写数据-flush
  1. 1.flush负责将内存中的segment写入磁盘,主要做如下的工作:
    -a.将translog写入到磁盘
    -b.将index buffer清空,其中的文档生成一个新的segment,相当于一个refresh操作
    -c.更新commit point并写入磁盘
    -d.执行fsync 操作,将内存中的segment写入磁盘
    -e.删除旧的translog文件

2.2 ES写性能优化

2.2.1 目的
  1. 1.目标是增大写吞吐量-EPS(Event Per Second)越高越好。
  2. 2.优化方案
    -客户端:多线程写,批量写
    -ES:在”高质量数据建模”的前提下,主要是在refresh、translog和flush之间做文章。
2.2.2 写性能优化-refresh
  1. 1.目标为降低refresh的频率
    -增大refresh_interval,降低实时性,以增大一次refresh处理的文档数,默认是1s,设置为-1直接禁止自动refresh
    -增大index buffer size,参数为indices.memory.index_buffer_size(静态参数,需要设定在elasticsearch.yxml中),默认为10%。
2.2.3 写性能优化-translog
  1. 1.目标是降低translog写磁盘的频率,从而提高写速率,但会降低容灾能力
    -index.translog.durability设置为async,index.translog.sync_interval设置需要的大小,比如120s,那么translog会改为每120s写一次磁盘
    -index.translog.flush_threshold_size默认为512mb,即translog超过该大小时会触发一次flush,那么调大该大小可以避免flush的发生。
2.2.4 写性能优化-flush
  1. 1.目标为降低flush的次数,在6.x可优化的点不多,多为es自动完成

ElasticStack-10-Elasticsearch篇之数据建模

本章会介绍使用 Elasticsearch 中要注意的数据建模常见问题以及优化思路和方案，让大家可以根据自己的业务场景设置最合理的模型。

1.数据建模简介

1.1 什么是数据建模?

  1. 1.英文为 Data Modeling,为创建数据模型的过程
  2. 2.数据建模(Data Model)定义:
    -对现实世界进行抽象描述的一种工具和方法
    -通过抽像的实体及实体之间联系的形式去描述业务规则,从而实现对现实世界的映射。

1.2 数据建模的过程

  1. 1.概念模型(10%)—确定需求
    -确定系统的核心需求和范围边界,设计实体和实体间的关系。不会讨论具体的属性的。
  2. 2.逻辑模型—根据需求,设计实体关系
    -进一步梳理业务需求,确定每个实体的属性、关系和约束等。
  3. 3.物理模型—确认使用哪个数据库产品做
    -结合具体的数据库产品,在满足业务读写性能等需求的前提下确定最终的定义。
    -Mysql、MongoDB、elasticsearch等。
    -第三范式
    物理模型解决的问题其实是最终需要落地的东西。

1.3 数据建模的意义


1.首先确认你的业务
2.其次你的系统数据支撑你的业务
3.系统数据需要怎样存储,就是需要考虑到数据模型了

2.ES数据建模配置相关介绍

2.1 ES的数据建模

  1. 1.ES是基于Lucene以倒排索引为基础实现的存储体系,不遵循关系型数据库中的范式约定

a.首先需要根据业务/数据需求来创建概念/逻辑模型,最后产出:实体列表、实体属性描述、实体键关系描述。
b.根据部署/性能需求产生物理模型,es中物理模型有ES Index Template、ES Index Mapping配置

2.2 Mapping字段的相关设置

  1. 1.enabled
    -true|false
    -当字段仅存储,不做搜索或聚合分析,把这个值设置为false,默认为true,设置为false,节省磁盘空间，比如我们做外部系统请求分析时候、有一些cookie、session数据一般不做查询分析,此时把设置为false

  2. 2.index
    -true|false
    -设置是否构建倒排索引

  3. 3.index_options
    -docs|freqs|positions|offsets
    -如果构建倒排索引的话,需要存储哪些信息。

  4. 4.norms
    -true|false
    -是否存储归一化相关参数,如果字段仅用于过滤和聚合分析,可关闭

  5. 5.doc_values
    -true| false
    -是否启用doc_values,用于排序和聚合分析时候的场景

  6. 6.field_data
    -false|true
    -是否为text类型启用fielddata,实现排序和聚合分析

  7. 7.store
    -false|true
    -是否存储此字段值,默认为false,因为默认存放在source下

  8. 8.coerce
    -true|false
    -是否开启自动数据类型转换功能,比如字符串转为数字,浮点转为整数等。
  9. 9.multifields多字段
    -灵活使用多字段特性来解决多样的业务需求

2.3 Mapping的其他设置

  1. 1.dynamic
    -true|false|strict
    -控制mapping自动更新

  2. 2.data_detection
    -true|false
    -是否自动识别日期类型

2.4 Mapping字段属性的设定流程


一个设定的流程如下：
1.是何种类型 2.是否需要检索(index设定) 3.是否需要排序和聚合分析 4.是否需要另行存储

2.4.1 是何种类型?
  1. 1.字符串类型
    -需要分词则设定为text类型,否则设定为keyword类型
  2. 2.枚举类型
    -基于性能考虑将其设置为keyword类型,即便该数据为整型。
  3. 3.数值类型
    -尽量选择贴近的类型,比如byte即可标识所有数值时,即选用byte,不要用long
  4. 其他类型
    -比如布尔类型、日期、地理位置数据等。
2.4.2 是否需要检索?
  1. 1.完全不需要检索、排序、聚合分析的字段
    -enabled设置为false
  2. 2.不需要检索的字段
    -index设置为诶false
  3. 3.需要检索的字段，可以通过如下配置设定需要的存储粒度
    -index_options结合需要设定
    -norms不需要归一化数据时关闭即可
2.4.3 是否需要排序和聚合分析?
  1. 1.不需要排序或者聚合分析 功能
    -doc_vales设定为false
    -fielddata设定为false
    2.4.4 是否需要另行存储?
  2. 1.是否需要专门存储当前字段的数据?
    -store设定为true,即可存储该字段的原始内容(与_source中的不相关)
    -一般结合_source的enabled设定为false时使用

  2. 2.通过设置index.mapping.total_fields.limit可以限定索引中最大字段数、默认是1000
  3. 3.如果字段确实比较多,可以通过key/value的方式解决字段过多问题,但是并不完美。

  1. 1.虽然通过这种方式可以极大地减少Field数目,但也有一些明显的坏处
    -query语句复杂度飙升,且有一些可能无法实现,比如聚合分析相关的
    -不利于在Kibana中做可视化分析

ElasticStack-9-Elasticsearch篇之聚合分析入门

介绍Elasticsearch 聚合分析的功能，让大家了解其分类、组成，带领大家逐个了解、掌握每一个聚合 API 的使用方法和技巧，为后续 Kibana 使用打好基础。

1.聚合分析简介

1.1 什么是聚合分析?

  1. 1.搜索引擎用来回答如下问题:
    -请告诉我地址为上海的所有订单?
    -请告诉我最近1天内创建但没有付款的所有订单?

  2. 2.聚合分析可以回答如下问题:

我们获取job前5个值,当es接收到查询语句后,coordinating Node会去获取每一个分片上的top 5 from Px，然后coordinating Node算出Top(p0,p1,p2,p3,p4)最小值返回给用户.上面执行流程中会不会有问题，Terms的执行流程并不永远准确。

我们以以下为例:我们拿两个节点为例:node1、node2。
p0上:a有5个 b有4个 c有3个 d有4个
p1上:a有5个 b有2个 c有3个 d有1个

7.文档分布式存储

  1. 1.文档最终会存储到分片上,如下图所示:
    -Document1最终存储在分片P1上,则在其副本R1上也存了一份

  2. 2.Document1是如何存储到分片P1的?选择P1的依据是什么?
    -Document1并不是随便选择到P1的,而是有一个文档到分片的映射算法。
    -映射算法目的:使得文档均匀分布在所有分片上,以充分利用资源。

  3. 3.算法
    -随机选择或者round-robin算法? 不可取:这种方法存储时候,文档存进去之后,我们需要读取的，比如：document1存储到p1了，我要通过这个文档id=1，如果是随机存储或者round-robin存储的获取，你是没法知道此文档1存储到那个分片上，两种选择:a.请求过来之后,在所有分片上都请求一下,有就返回,加入分片数有100个,浪费性能。b.开辟一个存储空间,存储文档到分片的映射关系。成本巨大(es是支持pb级别的数据存储,加入有上亿个文档,存储对应关系成本太高)。
    -我们采用的是：根据文档值试试计算对应的分片。

7.1 文档到分片的映射算法

  1. 1.es通过如下的公式计算文档对应的分片
    -shard = hash(routing)%number_of_primary_shards
    -hash算法保证可以将数据均匀地分散在分片中
    -routing是一个关键参数,默认是文档id,也可以自行指定。
    -number_of_primary_shards是主分片数。

  2. 2.该算法与主分片相关,这也是分片数一旦确定后便不能更改的原因。

8.脑裂问题

  1. 1.脑裂问题,英文为split-brian,是分布式系统中的经典问题,如下图所示:
    -3个节点组成的集群,突然node1的网络和其他两个节点中断.node2与node3会重新选举master.比如node2成为了新master,此时会更新cluster state[cluster state 3’]
    -node1发现与其他节点连接不上时候,node1自己组成集群后,也会更新cluster state

  2. 2.同一个集群中有两个master,而且维护不同的cluster state,网络恢复后无法选择正确的master

  3. 3.解决方案为仅在可选举master-eligible节点数大于等于quorum时才可以进行master选举

5.相关性算法

  1. 1.相关性算分是指文档与查询语句间的相关度,英文为:relevance
    -通过倒排索引可以获取与查询语句相匹配的文档列表,那么如何将符合用户查询需求的文档放到前列呢?
    -本质是一个排序问题,排序的依据是相关性算分。

  2. 2.相关性算分的几个重要概念如下:
    -Term Frequency(TF)词频,即单词在该文档中出现的次数,词频越高,相关性越高。
    -Document Frequency(DF)文档频率,即单词出现的文档数。
    -Inverse Document Frequency(IDF)逆向文档频率,与文档频率相反,简单理解为:1/DF,即单词出现的文档数越少,相似度越高。这个词在所在文档越少,说明这个单词越重要。
    -Field-length Norm 文档越短,相似度越高。

  3. 3.ES目前主要有两个相关性算分模型,如下：
    -TF/IDF模型
